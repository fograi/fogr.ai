  <action>
    Read `src/cron-worker.ts` fully. Locate the `findMatchingAds` function (research says lines 343-358).

    **The bug:** The function builds a Supabase REST URL with query params like:
    ```
    url.searchParams.set('county', `eq.${search.county}`);
    url.searchParams.set('locality', `eq.${search.locality}`);
    ```
    These query flat columns `county` and `locality` which DO NOT EXIST on the ads table. Location data is stored in the JSONB column `location_profile_data`.

    **The fix:** Replace with PostgREST JSONB path syntax:
    ```typescript
    // BEFORE (broken):
    if (search.county) url.searchParams.set('county', `eq.${search.county}`);
    if (search.locality) url.searchParams.set('locality', `eq.${search.locality}`);

    // AFTER (correct):
    if (search.county) {
      url.searchParams.set('location_profile_data->county->>id', `eq.${search.county}`);
    }
    if (search.locality) {
      url.searchParams.set('location_profile_data->locality->>id', `eq.${search.locality}`);
    }
    ```

    This matches the JSONB structure where ads have:
    ```json
    {
      "location_profile_data": {
        "county": { "id": "ie/leinster/dublin", "name": "Dublin" },
        "locality": { "id": "ie/leinster/dublin/clondalkin", "name": "Clondalkin" }
      }
    }
    ```

    Existing indexes `ads_location_county_id_idx` and `ads_status_county_created_idx` already support this query pattern (verified in research).

    **Also verify these are correct in the function:**
    1. The `status` filter uses `eq.active` (correct)
    2. The `category` filter uses `eq.{category}` on the `category` column (correct -- category IS a flat column)
    3. The `query` (keyword) filter uses text search if present
    4. The `created_at` filter uses `gt.{last_notified_at}` to only find new ads since last digest
    5. The `select` columns include the fields needed for the email template (id, title, slug, price, currency, image_keys)

    **Unsubscribe compliance check:**

    - Calls `isEmailSuppressed(env, userEmail, 'search_alerts')` before sending
    - Includes `buildUnsubscribeHeaders` in the sendEmail call
    - These were built in Phase 3 and should already be correct -- just verify they're in place
