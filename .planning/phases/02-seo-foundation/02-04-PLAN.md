<objective>
Update the ad page to handle expired ads gracefully: show full content publicly with a noindex directive, display similar active listings, hide the message composer, and return 410 Gone after 90 days post-expiry.

Purpose: Expired ad pages have inbound links from search engines and social shares. Returning 404 wastes that link equity and frustrates users. Showing the content with similar listings converts expired-ad visitors into active buyers.

Output: Modified ad page server load and component. Expired ads become publicly visible (reversing the current owner-only access check).
</objective>

The current ad page load function (line 39-49) blocks non-owner access to expired ads with `if (!isOwner) throw error(404, 'Ad not found')`. This must change per research Pitfall 3.

**New access control logic:**
Replace the existing `isNonPublic` check with:
```
const isExpired = ad.expires_at && ad.expires_at <= nowIso;
const isModRemoved = ad.status !== 'active' && ad.status !== 'expired';

// 410 Gone: expired more than 90 days ago
if (isExpired && ad.expires_at) {
  const daysSinceExpiry = (Date.now() - new Date(ad.expires_at).getTime()) / (1000 * 60 * 60 * 24);
  if (daysSinceExpiry > 90) {
    throw error(410, 'This ad has been removed');
  }
}

// Moderation-removed ads: owner-only
if (isModRemoved) {
  const { data: { user: authUser } } = await locals.supabase.auth.getUser();
  user = authUser;
  isOwner = !!user && user.id === ad.user_id;
  if (!isOwner) throw error(404, 'Ad not found');
}

// Expired ads: publicly visible (no login check needed for viewing)
// But still check auth for owner features
```

The key change: expired ads are no longer blocked for non-owners. Only moderation-removed ads (`rejected`, `removed`, `pending`) remain owner-only.

**Similar listings query (when expired):**
When `isExpired` is true, run a secondary Supabase query:
```
// Fetch similar active listings: same category + same county, fallback to category-only
let similarAds: AdCard[] = [];
if (isExpired) {
  const countyId = locationProfileData?.county?.id;

    .eq('status', 'active')
    .gt('expires_at', nowIso)
    .eq('category', ad.category)
    .not('id', 'eq', ad.id)
    .limit(6);

  // If county is available, prefer county-filtered results
  if (countyId && similar && similar.length >= 3) {
    // We got enough with just category, but let's try with county filter
    // Actually: query with county first, fall back to category-only if < 3 results
    const { data: countyMatches } = await locals.supabase
      .from('ads')
      .select('id, slug, title, price, image_keys, description, category, currency, category_profile_data, location_profile_data, firm_price, min_offer')
      .eq('status', 'active')
      .gt('expires_at', nowIso)
      .eq('category', ad.category)
      .not('id', 'eq', ad.id)
      .limit(6);
    // Note: filtering by county in location_profile_data JSONB is complex.
    // Simpler approach: fetch category-only results (limit 6) and that's sufficient.
  }

  // Map to AdCard format
  similarAds = (similar ?? []).map(s => ({
    id: s.id,
    slug: s.slug ?? undefined,
    title: s.title,
    price: s.price ?? null,
    img: s.image_keys?.[0] ?? '',
    description: s.description ?? '',
    category: s.category ?? '',
    categoryProfileData: (s.category_profile_data as Record<string, unknown> | null) ?? null,
    locationProfileData: (s.location_profile_data as Record<string, unknown> | null) ?? null,

  }));
```

**Simplification note:** Filtering by county within JSONB is complex without a dedicated column. For v1, just fetch same-category active ads (limit 6, excluding the expired ad itself). This gives relevant results without complex JSONB querying. If the category has very few listings, that is fine -- the empty state will show "No similar listings found."
