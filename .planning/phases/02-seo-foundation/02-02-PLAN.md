  <action>
Create `src/params/` directory (if it does not exist).

**src/params/category.ts:**
```typescript
import { slugToCategory } from '$lib/category-browse';
import type { ParamMatcher } from '@sveltejs/kit';

export const match: ParamMatcher = (param) => {
  return slugToCategory(param) !== null;
};
```

This ensures only valid category slugs (`home-garden`, `electronics`, `baby-kids`, `bikes`, `clothing-accessories`, `services-gigs`, `lessons-tutoring`, `lost-found`, `free-giveaway`) match the `[category=category]` route parameter. All other strings (like `about`, `login`, `ad`, `privacy`, `terms`) fall through to their static routes.

**src/params/county.ts:**
```typescript
import { getCountyOptions } from '$lib/location-hierarchy';
import slugify from 'slugify';
import type { ParamMatcher } from '@sveltejs/kit';

const COUNTY_SLUGS = new Set(
  getCountyOptions().map(c => slugify(c.name, { lower: true, strict: true }))
);

export const match: ParamMatcher = (param) => {
  return COUNTY_SLUGS.has(param.toLowerCase());
};
```

This builds a Set of valid county slugs at module load time from the existing location hierarchy data. Note: current county options are Republic of Ireland only. Northern Ireland counties will be added in Phase 4 (TRST-02).

Verify no collision between category slugs and county slugs by inspecting both sets. Research confirmed zero collisions between the current category slugs (`home-garden`, `electronics`, etc.) and county slugs (`dublin`, `cork`, etc.).

  <action>
Create three programmatic page routes. All follow the same pattern: server load fetches listings from `/api/ads` with filters, page component renders a listings grid with SEO head tags.

**Shared patterns for all three routes:**
- Server load fetches from `/api/ads` with appropriate category/county_id filters
- Server load counts active listings and computes price range (min/max from results)
- Server load returns: `{ ads, category, categorySlug, countyName, countySlug, listingCount, priceRange: { min, max }, seo, page, nextPage }`
- The `seo` object includes title, description, canonical, and robots (set to `noindex` if listingCount < 3 per user decision)
- Meta description pattern for programmatic pages: "Browse {count} second-hand {category} listings for sale in {county} on Fogr.ai. Prices from EUR {min} to EUR {max}." -- adapt based on available data

**Route 1: Category-only** (`src/routes/(public)/[category=category]/`)
- `+page.server.ts`:
  - Use `slugToCategory(params.category)` to get display name
  - Fetch `/api/ads?category={displayName}&limit=24&page={page}`
  - Title: `Second-Hand {Category} for Sale in Ireland | Fogr.ai`
  - Support pagination via `?page=N` query param
  - Return listing count from API response or query a count separately
- `+page.svelte`:
  - H1: `Second-Hand {Category} for Sale in Ireland` (dynamic, per user decision)
  - svelte:head with title, meta description, canonical, robots (noindex if < 3 listings)
  - OG tags using the same pattern as Plan 01
  - Listings grid using `AdCard` component
  - Stats line showing listing count and price range
  - Pagination links
  - "No listings yet" empty state

**Route 2: County-only** (`src/routes/(public)/[county=county]/`)
- `+page.server.ts`:
  - Look up the county by matching the slug against `getCountyOptions()` -- build a map from slugified county name to county option {id, name}
  - Fetch `/api/ads?county_id={countyId}&limit=24&page={page}`
  - Title: `Second-Hand Classifieds in {County} | Fogr.ai`
- `+page.svelte`:
  - H1: `Second-Hand Classifieds in {County}`
  - Same pattern as category-only

**Route 3: Category+County** (`src/routes/(public)/[category=category]/[county=county]/`)
- `+page.server.ts`:
  - Validate both params, fetch with both filters
  - Title: `Second-Hand {Category} for Sale in {County} | Fogr.ai`
- `+page.svelte`:
  - H1: `Second-Hand {Category} for Sale in {County}`
  - Same pattern as above

**Important implementation details:**
- Use `slugify` (already installed) to create a reverse lookup map from county slug to county option `{id, name}`. Build this at module level (not per request).
- Export a helper function `countySlugToOption(slug: string): { id: string, name: string } | null` from the county-only page server load (or from a shared module) for reuse by the category+county route. Consider placing this in `src/lib/seo/county-slugs.ts` for reuse by sitemap (Plan 03).
- The page components should be simple and reusable. Consider creating a shared `ProgrammaticListingPage.svelte` component in `src/lib/components/` if the three pages are nearly identical, OR just duplicate with minor differences (3 small files is fine for clarity).
- OG fallback images: reference `/og-fallback/{categorySlug}.png` for category pages. These static files will not exist yet -- that is fine, the URL pattern is correct and the files can be added as static assets later.
- Filter chips: include active filter chips for the current category/county selection on the page (same pattern as existing category browse page).
