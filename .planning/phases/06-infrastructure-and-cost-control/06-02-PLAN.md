    **2. Create `.planning/ops/DEGRADATION.md`:**

    Document ALL existing graceful degradation paths and the R2 risk acceptance. Structure:

    **Purpose:** This document verifies and records the platform's existing graceful degradation behavior. Per the Phase 6 CONTEXT.md decision: "Verify and document these existing degradation paths rather than building new ones."

    **OpenAI Moderation Unavailable (INFR-04):**

    When OpenAI's moderation API is unreachable or returns errors:

    - **POST /api/ads (ad creation):** At line 557 of `src/routes/api/ads/+server.ts`, the status is set to `'pending'` when `moderationUnavailable` is true. The ad is saved to the database and images go to the pending R2 bucket. User sees "Ad submitted and pending review." -- no error is surfaced. The cron worker will pick up the ad for moderation when OpenAI recovers.
    - **Cron worker (batch moderation):** In `src/cron-worker.ts`, when `moderateText()`, `moderateTextAndImage()`, or `moderateSingleImage()` returns `'unavailable'`, the worker logs `cron_moderation_unavailable` and `continue`s to the next ad. The ad stays in pending state and will be retried on the next 15-minute tick.
    - **What this means for the user:** Ads still submit successfully. They just stay in "pending review" state until OpenAI becomes available again. No ads are rejected due to API unavailability.
    - **What triggers this:** OpenAI API key hits spending limit (hard limit), network errors, OpenAI service outage.
    - **Recovery:** Automatic. Once OpenAI responds again, the next cron tick processes the backlog.

    **Email Service Unavailable (INFR-04):**

    When Resend is unreachable or returns errors:

    - **All email sends:** The `sendEmail()` function in `src/lib/server/email/send.ts` uses a fire-and-forget pattern. Resend handles delivery retries natively on their infrastructure.
    - **Ad approval/rejection emails (cron worker):** Email sends happen AFTER status updates (Phase 3 decision: 03-03). If email fails, the ad is still approved/rejected correctly. The user just doesn't get notified by email.
    - **Message notification emails (API):** Sent via `platform.ctx.waitUntil()` -- non-blocking. If email fails, the message is still delivered in-app.
    - **What this means for the user:** Core platform functions (posting, browsing, messaging) continue working. Users may not receive email notifications until Resend recovers, but they can still check the platform directly.
    - **Recovery:** Automatic. Resend retries failed deliveries internally.

    **Database Unavailable:**

    - The platform does NOT have explicit graceful degradation for Supabase outages. If the database is unreachable, all data-dependent operations fail.
    - This is acceptable for v1: Supabase Pro has a 99.9% uptime SLA. A Supabase outage would be a total platform outage.
    - The `/api/health` endpoint will return `{ status: 'down' }` and 503, triggering UptimeRobot alerts.

    **R2 Storage Unavailable:**

    - Image uploads will fail with a 503 "Storage temporarily unavailable" message (line 308 in POST handler).
    - Existing ads with already-uploaded images will still display correctly (images served via CDN, which may have edge cache).
    - The `/api/health` endpoint will detect R2 unavailability and return `{ status: 'down' }`.

    **R2 Image Redundancy -- Risk Acceptance (INFR-05):**

    Per Phase 6 CONTEXT.md decision: "Documented risk acceptance -- R2's 11-nines durability is sufficient for v1."

    - Cloudflare R2 provides 99.999999999% (11 nines) data durability
    - At this durability level, backup/replication adds cost and complexity for negligible risk reduction
    - If Cloudflare has a catastrophic R2 failure (extraordinarily unlikely), images are lost but:
      - Ad metadata is preserved in Supabase database backups
      - Ads can be re-posted by sellers
      - This is a classifieds platform, not a photo archive -- images are transient by nature
    - R2 accessibility is monitored via the `/api/health` endpoint to catch binding misconfigurations or temporary outages early
    - This risk acceptance will be revisited if the platform scales to a point where image loss would affect a significant number of users

    **Summary Table:**

    | Service       | Failure Mode              | Platform Behavior                      | User Impact                    | Recovery     |
    |---------------|---------------------------|----------------------------------------|--------------------------------|--------------|
    | OpenAI        | API errors/spending limit | Ads queue as 'pending'                 | Delayed moderation, no errors  | Automatic    |
    | Resend        | API errors/outage         | Emails silently skipped                | No email notifications         | Automatic    |
    | Supabase      | Database outage           | All data operations fail               | Full outage                    | Wait for SLA |
    | R2            | Storage outage            | New uploads fail, existing images work | Cannot post new images         | Wait for SLA |
    | R2            | Catastrophic data loss    | Images permanently lost                | Sellers re-post affected ads   | Manual       |
