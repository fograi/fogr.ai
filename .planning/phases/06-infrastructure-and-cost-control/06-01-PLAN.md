  <action>
    **1. Create `src/routes/api/health/+server.ts`:**

    Export a single GET RequestHandler (public, no auth, no CSRF). The endpoint checks three things and returns JSON:

    ```typescript
    type HealthCheck = {
      status: 'ok' | 'degraded' | 'down';
      checks: {
        database: { ok: boolean; latency_ms?: number };
        cron: { ok: boolean; last_tick?: string; age_minutes?: number };
        r2: { ok: boolean };
      };
      timestamp: string;
    };
    ```

    **DB check:** Use `locals.supabase.from('ads').select('id').limit(1)` (anon client, no service role needed). Measure latency. `ok: !error`.

    **Cron heartbeat check:** Read KV key `cron:heartbeat` from `platform.env.RATE_LIMIT` (cast platform.env to `{ RATE_LIMIT?: KVNamespace }`). Parse the ISO timestamp, compute age in minutes. `ok` if age <= 30 minutes. If key is missing (first deploy or KV expired), `ok: false`.

    **R2 check:** HEAD request on key `_health-check` from `platform.env.ADS_BUCKET` (cast to `{ ADS_BUCKET?: R2Bucket }`). A `null` return (key doesn't exist) is SUCCESS -- it proves R2 responded. Only a thrown exception means R2 is unreachable. `ok: true` in try block, `ok: false` in catch.

    **Status logic:**
    - `down`: if DB or R2 is not ok (site functionality broken)
    - `degraded`: if cron is not ok but DB and R2 work (site usable, moderation delayed)
    - `ok`: all three checks pass

    **Response:** Return `json({ status, checks, timestamp }, { status: allOk ? 200 : 503, headers: { 'cache-control': 'no-store, no-cache, must-revalidate' } })`.

    The 503 on unhealthy is critical -- UptimeRobot detects non-2xx as down. The no-store cache-control prevents CDN caching stale health responses.

    Import types from `@cloudflare/workers-types` (KVNamespace, R2Bucket). Import `json` and `RequestHandler` from `@sveltejs/kit`.

    **2. Modify `src/cron-worker.ts`:**

    Add `RATE_LIMIT?: KVNamespace` to the existing `Env` type (import `KVNamespace` from `@cloudflare/workers-types` -- it's already imported for other types like R2Bucket).

    At the END of the `try` block in the `scheduled()` handler (after all existing task calls -- after `runSavedSearchDigest`, after all the `if` blocks), add the heartbeat write:

    ```typescript
    // Write heartbeat timestamp for /api/health staleness check
    if (env.RATE_LIMIT) {
      try {
        await env.RATE_LIMIT.put('cron:heartbeat', new Date().toISOString(), {
          expirationTtl: 3600 // auto-expire after 1 hour (safety net)
        });
        console.log('cron_heartbeat_written');
      } catch (err) {
        console.error('cron_heartbeat_failed', err);
      }
    }
    ```

    Place this INSIDE the existing try block but BEFORE the catch. The heartbeat confirms "cron ran to completion" -- it should only write if all tasks executed without throwing.

    **3. Update `wrangler.cron.jsonc`:**

    Add the KV namespace binding so the cron worker can access RATE_LIMIT. Add this after the `r2_buckets` array:

    ```jsonc
    "kv_namespaces": [
      {
        "binding": "RATE_LIMIT",
        "id": "cf5daa23362c48639599f07aa6afe7aa"
      }
    ]
    ```

    Use the SAME KV namespace ID as in `wrangler.jsonc` (`cf5daa23362c48639599f07aa6afe7aa`). No `preview_id` needed for the cron worker. No `remote: true` needed.
